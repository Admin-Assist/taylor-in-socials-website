# Taylor In Socials Site Map + Portal Wireframe Plan

## Product Shape

Taylor In Socials should be built as two connected experiences:

- Public marketing website: open, editorial, conversion-focused, designed to sell the service and capture free audit bookings.
- Client portal: protected, role-based, and operational, designed for existing clients to review plans, approvals, reports, access workflows, and billing status.

The public website should feel premium and calm. The portal should feel clear, trustworthy, and practical: the client should always understand what has happened, what is next, and what needs their input.

## Brand Direction

Visual tone:

- Cream base background.
- Dusty blush accents.
- Charcoal or soft black typography.
- Elegant serif headings.
- Clean sans-serif body copy.
- Editorial spacing.
- Restrained motion, mostly fade, slide, and subtle hover transitions.
- Logo-led composition, with the brand mark treated as a luxury boutique signal.

Design principles:

- Avoid template-heavy SaaS styling.
- Use generous whitespace but keep portal screens efficient.
- Use real client outcomes, screenshots, or content examples where possible.
- Keep the public side aspirational, and the portal side operational.

### Client Portal

Protected routes:

- `/portal`
- `/portal/content-plan`
- `/portal/approvals`
- `/portal/linked-accounts`
- `/portal/credentials`
- `/portal/reports`
- `/portal/billing`
- `/portal/settings`

Role model:

- Client owner: can view all account data, approvals, reports, billing status, and credential requests for their business.
- Client team member: can view plans, comment, and approve assigned items.
- Admin: can view and manage all clients.
- Strategist: can manage content plans, reports, and approvals for assigned clients.
- Finance/admin support: can view billing and retainer status without credential access unless explicitly granted.

Access rule:

- Every portal record must belong to a client account or workspace.
- Portal users should only see records where their user account is assigned to that client workspace.
- Admin roles can switch between client workspaces.

### Secure Credentials Workflow

Goal:

Avoid casual password sharing while still supporting real-world client onboarding.

Preferred approach:

- Use OAuth or platform-native delegated access where available.
- For unavoidable credentials, use a dedicated secure vault provider or encrypted credential workflow.
- Do not store plain text passwords in the application database.

Modules:

- Credential requests.
- Status: requested, submitted securely, verified, revoked, expired.
- Platform name.
- Access owner.
- Last verified date.
- Admin-only access notes.

Client-facing copy:

- Keep it short and reassuring.
- Explain that secure links or delegated access are preferred.
- Avoid showing passwords back to users.

## Core Data Model

Recommended entities:

- User
- Role
- ClientWorkspace
- WorkspaceMember
- ServicePlan
- ContentItem
- ContentAsset
- Approval
- Comment
- LinkedAccount
- CredentialRequest
- Report
- ReportMetric
- InvoiceStatus
- AuditLead
- Booking

Key relationships:

- A client workspace has many users.
- A user can belong to many client workspaces.
- A content item belongs to one workspace.
- Approvals and comments belong to one content item.
- Reports belong to one workspace and one reporting period.
- Linked accounts belong to one workspace.
- Credential requests belong to one workspace and one platform.

## Security Requirements

Baseline requirements:

- Real authentication, not a custom homemade login.
- Role-based access control.
- Workspace-level data isolation.
- Server-side authorization checks for every portal data request.
- Secure password reset.
- HTTPS in production.
- Audit trail for approvals and sensitive workflow events.
- No plain text password storage.
- Minimal access for admin roles.

Credential handling:

- Prefer OAuth/delegated access.
- Use a secure vault integration where credentials are unavoidable.
- Store only metadata in the portal database.
- Never display submitted passwords back to users.

## Booking Flow

Free audit path:

1. Visitor clicks "Book your free social + web reach audit".
2. Visitor lands on `/free-audit`.
3. Visitor reviews what is included.
4. Visitor books through embedded calendar.
5. Booking form collects:
   - Name.
   - Email.
   - Business name.
   - Website URL.
   - Social handles.
   - Main goal.
6. Booking creates or syncs an audit lead record.
7. Admin receives notification.
8. After call, lead can be converted into a client workspace.
